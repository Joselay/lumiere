## Objective
Build a deterministic Python Telegram bot that can automatically trade BTC on OKX demo trading using `okxapi/python-okx`.

Reference SDK cached locally:
`/Users/menglay/.cache/checkouts/github.com/okxapi/python-okx`

## Initial scope
- Exchange: OKX demo environment only
- Market focus: Bitcoin only
- Language: Python
- Telegram bot interface for status, controls, and notifications
- Fully automated trading loop; user should not need to manually place trades
- No GPT/Claude/LLM-based trade decisions
- Trading decisions must come from deterministic, testable strategy rules

## Modern library choices to use first
Start with this minimal modern Python stack before adding strategy complexity:

1. `uv` — project/dependency management, lockfile, running scripts, reproducible setup.
2. `pydantic-settings` — typed environment/config loading for OKX demo credentials, Telegram token, risk limits, and strategy params.
3. `aiogram` 3.x — async-first Telegram bot framework with routers/middleware; better fit for a bot that also runs a background trading loop.
4. `structlog` — structured JSON-ish logs for auditability of strategy inputs, decisions, orders, fills, and errors.
5. `pytest` + `pytest-asyncio` — tests for async Telegram handlers, strategy rules, risk limits, and engine loop.
6. `ruff` — formatter/linter to keep the codebase simple and consistent.

Keep `okxapi/python-okx` as the OKX integration library because it is the referenced official/community SDK for this todo. If its calls are synchronous, isolate them behind an OKX client wrapper and call them from async code using `asyncio.to_thread` where needed.

Avoid adding heavier libraries at first:
- No database/ORM until trade history persistence is needed.
- No scheduler framework until a plain `asyncio` loop is insufficient.
- No pandas/numpy until a chosen strategy needs larger historical indicator calculations.
- No ML/LLM libraries for trade decisions.

## Required capabilities
- Configure OKX demo API credentials securely via environment variables / local config excluded from git
- Connect to OKX demo account using `python-okx`
- Fetch BTC market data and account balances/positions
- Generate trade decisions from deterministic strategy module, e.g. moving average crossover, RSI, breakout, or trend-following rules
- Place, monitor, and cancel demo orders automatically
- Telegram commands for:
  - `/start` bot health check
  - `/status` current account/position state
  - `/strategy` show active deterministic strategy and parameters
  - `/pause` stop automated trading
  - `/resume` resume automated trading
  - `/panic` cancel open orders and stop trading
- Send Telegram alerts for strategy signals, orders, fills, errors, and risk-limit events

## Safety/risk controls, even in demo
- Demo-only guard to prevent accidental live trading
- Trade BTC only allowlist
- Max position size
- Max daily loss / drawdown limit
- Cooldown between trades
- Stop trading after repeated API/order failures
- Strategy decision logging so every trade can be explained by exact input values/rules
- Structured logs for auditability

## Suggested implementation phases
1. Initialize the project with `uv`, `pyproject.toml`, `ruff`, and `pytest`
2. Add typed config with `pydantic-settings`
3. Add OKX demo client wrapper around `python-okx`
4. Add Telegram bot command layer with `aiogram` 3.x
5. Add deterministic strategy interface
6. Implement first BTC strategy, preferably simple and testable
7. Add automated trading engine loop using `asyncio`
8. Add risk manager and demo-only guard
9. Add tests/mocks for OKX, Telegram, strategy decisions, and risk limits
10. Run OKX demo validation before any live expansion

## Acceptance criteria
- Bot can run locally in OKX demo mode without live trading permissions
- Bot only trades BTC instruments
- No LLM/GPT/Claude dependency is required for trade decisions
- Strategy output is deterministic for the same market/account inputs
- User can pause/resume/panic-stop from Telegram
- Orders, strategy signals, and errors are reported in Telegram
- Automated loop respects configured risk limits

## Reference review: okxapi/okx-sample-market-maker 2026-06-23
Cached and reviewed `/Users/menglay/.cache/checkouts/github.com/okxapi/okx-sample-market-maker` at `f726dc4`.

Useful lessons:
- Shows OKX demo/paper flag convention (`flag='1'`) and strong recommendation to use demo trading.
- Confirms batch order/cancel/amend limits around 20 orders per REST batch.
- Provides useful order request field mapping (`instId`, `tdMode`, `side`, `ordType`, `sz`, `clOrdId`, `tag`, `posSide`, `px`, `reduceOnly`, `tgtCcy`).
- Provides instrument utilities for inst type detection, tick-size price trimming, and lot-size quantity trimming; good future improvement before more advanced strategies.
- Provides health-check ideas: stale order book/account data detection, OKX maintenance status checks, checksum validation, and cancel-all on strategy exceptions.
- Provides WebSocket architecture ideas for low-latency order/account/orderbook caches.

Caveats:
- It is a market-maker sample (`BTC-USDT-SWAP`, cross margin, many limit orders), not the current long-only/simple BTC demo bot scope.
- It stores credentials in `settings.py`; our project should keep using `.env` via `pydantic-settings`.
- It depends on heavier/older stack (`Twisted`, `autobahn`, `PyYAML`) that we intentionally avoided initially.
- Its websocket imports use old modules (`okx.websocket.WsPrivate`, `WsPublic`) that are not present in current `python-okx 0.4.1`; latest SDK uses `WsPrivateAsync` / `WsPublicAsync` and demo URLs like `wss://wspap.okx.com:8443/ws/v5/...`.

Action taken from review:
- Improved our OKX account snapshot parsing so spot BTC balances from account details count as BTC position. Without this, `BTC-USDT` cash mode could repeatedly think the account is flat because spot holdings are not returned as derivatives positions.
- Added parser tests. Verification now: `uv run pytest -q` passes 14 tests; `uv run ruff check .` passes.
