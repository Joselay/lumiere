## Why
The engine currently records submitted orders but not confirmed fills into `AttributionLedger`. `/performance` can show zero attributed trades even after orders, and realized slippage is unavailable.

## Scope
- After order submission, poll/query OKX fills using `python-okx==0.4.1` and match by order/client order id.
- Persist `fill` events with decision price, fill price, fee currency, order id, trade id, and latency.
- Reconcile partial fills, cancellations, rejected orders, and duplicate fill rows.
- Compute realized slippage, fee attribution, daily PnL, and fill completeness alerts from actual fills.
- Increase fill-history pagination beyond the current single 100-row window so active days cannot silently miss fills.

## Ready when
Attribution PnL and OKX fill-derived PnL agree within fees/rounding, and promotion evidence fails if any order lacks final fill/reject/cancel attribution.

## Completion evidence
- Added `OKXDemoClient.fetch_order_fills()` using the python-okx Trade API, matching rows by `ordId`/`clOrdId`, preserving decision price, fee currency, trade id, raw row, and latency.
- Paginated OKX fill queries past one 100-row response for both current fills and fills history, with deduplication across endpoints.
- Attribution reports now dedupe duplicate fill rows, compute realized slippage from actual fill events, expose fill-completeness metrics, and alert when orders lack full fill/reject/cancel attribution.
- Added tests for OKX order-fill reconciliation, fill-history pagination, duplicate fill de-duping, missing order attribution alerts, and existing live attribution/PnL paths.
- Verification: `uv run ruff check .` and `uv run pytest -q`.
