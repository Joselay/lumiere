## Why
`lumiere-evidence` creates a packet, but readiness still depends on a human remembering the full runbook. It does not yet enforce observation days, consecutive gate passes, data length, fill completeness, slippage bounds, API error clusters, or stage-specific rollback checks.

## Scope
- Encode every `docs/go_live_runbook.md` stage threshold as machine-checkable rules.
- Require optimizer/backtest artifacts, paper ledger, attribution ledger, fill reconciliation, observation duration, and consecutive gate history as applicable.
- Fail if accepted candidates are stale, data is too short, fills are incomplete, alerts exist, or current env differs from accepted config.
- Output a concise go/no-go report with blockers sorted by severity.
- Add CI-style tests with passing and failing fixture packets.

## Ready when
We can run one command before promotion and get a defensible `go=false` until every readiness blocker is cleared.

## Completion evidence
- Reworked `uv run lumiere-evidence` into a strict readiness auditor with severity-sorted blockers and non-zero no-go exit.
- Added machine checks for required optimizer/backtest artifacts, stale accepted candidates, historical data length, backtest/OOS thresholds, paper/demo observation duration, attribution metrics, fill reconciliation, alerts, slippage bounds, consecutive gate history, API error clusters, runtime config matching, and larger-demo approval/risk controls.
- Updated runbook/README usage for the strict audit command.
- Added CI-style passing/failing readiness fixtures in `tests/test_readiness_auditor.py` and refreshed evidence CLI smoke/unit coverage.
